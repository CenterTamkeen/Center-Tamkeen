# منصة "تمكين" التعليمية (Tamkeen E-Learning Platform)

> منصة تعليمية متكاملة متعددة المستأجرين (Multi-Tenant) لسنتر "تمكين"، تتيح استضافة مجموعة من المدرسين في مكان واحد، مع لوحات تحكم منفصلة لكل من الإدارة المركزية والمدرسين، وطلب اشتراك يدوي حاليًا مع تجهيز معماري لإضافة بوابة دفع لاحقًا.

---

## وثيقة متطلبات العمل الرسمية (Business Requirements Document - BRD)

### 1. نظرة عامة على المشروع (Project Overview)

بناء منصة تعليمية متكاملة لسنتر "تمكين"، تتيح استضافة مجموعة من المدرسين في مكان واحد بنظام (Multi-Tenant). يمتلك كل مدرس صفحة مخصصة ولوحة تحكم لإدارة محتواه وأرباحه، بينما تمتلك إدارة سنتر "تمكين" (الأدمن) لوحة تحكم مركزية لإدارة النظام بالكامل، مراجعة العمليات المالية، وتفعيل المحتوى للطلاب.

تم تصميم المنصة بهندسة برمجية مرنة: الطلبات تُنشأ بحالة `pending` وتُفعّل عند تحولها إلى `completed`، سواء تم ذلك يدويًا من الأدمن حاليًا أو عبر Webhook دفع مستقبلاً.

---

### 2. التقنيات المستخدمة (Tech Stack)

تم اختيار هذه التقنيات لضمان أعلى كفاءة وأداء، وتأمين المحتوى بأقل تكلفة تشغيلية:

| الطبقة | التقنية | السبب |
| --- | --- | --- |
| **Full-Stack Framework** | Next.js (App Router) | سرعة التصفح وتوافق الموقع مع محركات البحث (SEO) |
| **Styling** | Tailwind CSS | واجهات احترافية وسريعة الاستجابة على كل الشاشات |
| **اللغة والاتجاه** | عربي بالكامل + RTL | المنصة كلها باللغة العربية واتجاه الكتابة من اليمين لليسار (RTL) كأساس |
| **Database** | Supabase (PostgreSQL) + RLS | عزل البيانات وحمايتها على مستوى الصفوف |
| **Authentication** | Supabase Auth | تأمين حسابات الطلاب والمدرسين |
| **Hosting** | Vercel | استقرار المنصة وسرعة استجابة الخوادم |
| **Video Streaming & Security** | Bunny Stream + signed embed tokens | رفع الفيديو من لوحة المدرس وتشغيله عبر روابط مؤقتة بعد التحقق من صلاحية الدرس |

### 3. المتطلبات الوظيفية (Functional Requirements)

#### أولاً: واجهة الطلاب والمستخدمين (Student Storefront)

- **تسجيل الطالب (Student Registration):** فورم تسجيل يأخذ بيانات الطالب الكاملة مع فالديشن: الاسم رباعي، رقم تليفون الطالب، رقم تليفون الأب، اسم المدرسة، النوع (ذكر/أنثى)، السنة الدراسية (أولى/تانية/تالتة ثانوي)، الشعبة (مرتبطة بالسنة)، الإيميل، الباسورد وتأكيده، وصورة الطالب.
- **الصفحة الرئيسية (Homepage):** صفحة تسويقية خاصة بسنتر "تمكين" تعرض أبرز المدرسين، أحدث الكورسات، وآراء وتقييمات الطلاب.
- **صفحة استكشاف المواد (Explore):** تتيح للطلاب البحث وتصفية الكورسات (تصفية باسم المادة، اسم المدرس، السعر).
- **الصفحات الشخصية للمدرسين (Teacher Profiles):** رابط ديناميكي لكل مدرس يعرض صورته، والنبذة التعريفية عنه، والكورسات الخاصة به فقط.
- **بوابة الطالب والاشتراك الحالي (Student Portal & Course Request):**
  - الطالب يطلب الاشتراك في كورس واحد من صفحة تفاصيل الكورس، مع إمكانية تطبيق كوبون.
  - صفحة "كورساتي" في بوابة الطالب تعرض المواد المفعلة من `enrollments`.
  - تشغيل الفيديو يتم عبر Bunny Stream بعد التحقق من أن الحصة Preview أو أن الطالب مشترك في الكورس.

#### ثانياً: آلية الدفع وإتاحة الكورسات (Payment & Order Lifecycle)

يعتمد النظام الحالي على **طلب اشتراك يدوي** لحين حسم تكامل الدفع:

- **دورة الطلب الحالية:**
  - الطالب يطلب الكورس من صفحة التفاصيل.
  - يتم إنشاء `order` بحالة `pending` مع `order_items`.
  - الأدمن يقبل الطلب يدويًا من لوحة الإدارة، فتتحول الحالة إلى `completed`.
  - Trigger قاعدة البيانات يفتح الكورس تلقائيًا في `enrollments` ويضيف أرباح المدرس في `teacher_earnings`.
  - يمكن رفض الطلب مع حفظ `rejection_reason`.

#### ثالثاً: لوحة تحكم المدرس (Teacher Dashboard)

نظام معزول يعتمد على صلاحيات المستخدم (Role-Based Access)، يتيح للمدرس:

- **إدارة المحتوى:** رفع الفيديوهات وترتيب الحصص داخل الكورسات الخاصة به وتحديد أسعارها وإصدار كوبونات خصم.
- **الإحصائيات المعتمدة:** لوحة مبسطة تعرض عدد الطلاب وإجمالي الأرباح المستحقة للمدرس (يتم تحديثها فقط بعد تحول الطلب إلى `completed`).

#### رابعاً: لوحة تحكم الإدارة (Admin Dashboard - إدارة تمكين)

مخصصة لإدارة سنتر "تمكين" بصلاحيات مطلقة:

- **إدارة المدرسين والمحتوى:** إضافة وحساب المدرسين، وتوليد روابطهم التلقائية (Slugs)، ومراجعة الكورسات.
- **متابعة الطلبات (Orders Monitor):**
  - واجهة لاستعراض كل الطلبات وحالتها (`pending` / `completed` / `rejected`).
  - قبول الطلب يدويًا لتفعيل الكورس، أو رفضه مع سبب الرفض.
- **التقارير المالية:** متابعة حركة المبيعات الكلية، الكوبونات المستخدمة، وأرباح سنتر "تمكين".

### 4. متطلبات الأمان وعزل البيانات (Security & Architecture)

- **حماية الفيديوهات:** رفع الفيديوهات إلى Bunny Stream من لوحة المدرس، وإصدار رابط تشغيل مؤقت من `/api/bunny/playback` بعد التحقق من صلاحية `lessonId` عبر RLS. يتطلب الإنتاج تفعيل Embed View Token Authentication وضبط `BUNNY_STREAM_TOKEN_SECURITY_KEY`.
- **عزل بيانات المدرسين:** تفعيل Row Level Security (RLS) في قاعدة البيانات لضمان عدم قدرة أي مدرس على التعديل أو الوصول لبيانات وكورسات مدرس آخر.
- **حماية الجلسات والمسارات:** استخدام HttpOnly Cookies والـ Middleware لتأمين مسارات النظام (Routes) والتحقق من صلاحيات الطلاب والتحقق من حالة دفع الطلب قبل عرض محتوى الكورسات.


## 🛠️ حالة الطلبات الحالية وتجهيز الدفع المستقبلي

الكود الحالي لا يحتوي تكامل فوري فعلي ولا Webhook دفع. الموجود هو دورة طلبات يدوية مبنية بحيث يسهل ربط أي بوابة دفع لاحقًا:

### 1. بنية جدول الطلبات (Orders Schema)

تم إنشاء جدول الـ `orders` في Supabase ليتعامل مع "حالة الطلب" (Order Status) كحالة مجردة:

- **حقول الجدول:** `id`, `student_id`, `total_amount`, `status` (`pending` | `completed` | `rejected`), `fawry_ref_no`, `rejection_reason`.
- عند طلب الكورس: يتم إنشاء طلب بحالة `pending`.
- عند قبول الأدمن: تتحول الحالة إلى `completed`.

### 2. فصل منطق تفعيل الكورس (Decoupled Activation Logic)

تم تصميم الباك إند عبر (Postgres Trigger) بحيث أن الكورس يُفتح للطالب وتُحدَّث أرباح المدرس بمجرد تحول حالة الطلب في قاعدة البيانات إلى `completed`؛ هذا يفصل منطق التفعيل عن مصدر التأكيد، ويسهّل إضافة أي بوابة دفع إضافية مستقبلاً دون تعديل منطق التفعيل.

### 3. تكامل الدفع لاحقًا

يمكن إضافة مسار Webhook لاحقًا لأي بوابة دفع. وظيفة الـ Webhook المطلوبة ستكون فقط التحقق من التوقيع ثم تحديث `orders.status = completed`، وبعدها تعمل Triggers قاعدة البيانات الحالية لفتح الكورس وحساب الأرباح.
